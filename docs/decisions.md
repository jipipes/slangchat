# SlangChat 설계 결정 기록

프로젝트 진행 중 내린 주요 판단과 그 근거를 시간순으로 남긴다. 각 항목은 문제 상황 → 결정 → 근거 순으로 적는다.

---

## 2026-07-20: 자동 슬랭 수집기 대신 수작업 큐레이션 채택

**문제**: Reddit/Twitter/Urban Dictionary를 크롤링해서 슬랭 데이터를 자동 수집하려 했음.

**결정**: 자동 크롤러는 만들지 않고, 수작업 입력(+ 필요시 LLM 초안 보조)으로 전환.

**근거**: Twitter/X는 2023년 API 유료화 이후 무료 크롤링이 사실상 불가능. Reddit도 API 정책상 대량 크롤링에 제약이 있고, AI 학습 목적 사용은 별도 라이선스가 필요. Urban Dictionary는 크롤링이 ToS 위반이며 데이터 품질(장난성 정의 다수)도 낮음.

## 2026-07-20: 언어 스코프 4개 → 2개로 축소

**문제**: 한국어/영어/일본어/중국어 4개 언어를 동시에 수집, 검증하려 했음.

**결정**: 한국어+영어로 축소. 스키마의 `Language` enum에 값만 추가하면 되도록 확장 가능한 구조는 유지.

**근거**: 4개 언어 동시 수작업 큐레이션은 3주 일정 대비 리소스 초과. 품질을 우선하고 언어는 나중에 늘리는 게 합리적.

## 2026-07-20: 파인튜닝 제외, RAG(사전+시맨틱 검색) 방식 채택

**문제**: Qwen2.5-1.5B를 QLoRA로 파인튜닝해 슬랭 이해력을 높이려 했음.

**결정**: 파인튜닝을 하지 않고 사전 매칭 + 벡터 검색 하이브리드로 대체. LLM은 답변 생성에만 사용.

**근거**: 400~600건 규모 데이터로는 파인튜닝 시 과적합 위험이 크고, 애초에 슬랭 "탐지"는 사전+검색만으로 충분히 해결 가능한 문제라 목적 대비 과설계였음.

## 2026-07-21: `.venv-py39-backup` 폴더 커밋 실수 제거

**문제**: 파이썬 가상환경 폴더 전체(bin, site-packages 포함)가 git 레포에 커밋됨.

**결정**: `git rm -r --cached`로 추적 해제, `.gitignore`에 `.venv*/` 추가.

**근거**: 가상환경은 로컬에서 언제든 재생성 가능한 산출물이라 커밋 대상이 아님. 용량만 키우고 다른 사람 클론 시 무의미함.

## 2026-07-21: 한국어 강조 접두사 화이트리스트를 완전/개/핵으로 제한

**문제**: 존/씨 등도 강조 접두사 후보로 고려했음.

**결정**: `KOREAN_EMPHASIS_PREFIXES`를 완전/개/핵만 포함하도록 제한.

**근거**: 존/씨를 접두사로 일반화하면 "존킹받네", "씨킹받네"처럼 실제로 쓰이지 않는 조합에서도 오탐 발생. 테스트로 확인된 생산적 접두사만 화이트리스트에 남기고, 나머지 표현(예: 존맛)은 사전에 개별 표제어로 등록하는 방식으로 처리.

## 2026-07-21 — "존버" 표제어를 "존버하다"로 수정

**문제**: 형태소 분석기(Kiwi) 기반 활용형 탐지가 "존버하면" 같은 문장에서 작동하지 않음.

**결정**: 용언형 슬랭은 항상 "다"로 끝나는 기본형으로 `term`을 등록하는 규칙을 확립. "존버" → "존버하다"로 수정.

**근거**: `KiwiSlangDetector`는 `term.endswith("다")`인 항목만 동사 어간으로 등록해 활용형을 자동 탐지하도록 설계되어 있음. 명사형으로 등록하면 이 경로를 타지 못해 활용형을 놓침. 실제 테스트(`존버하면 이긴다` → 탐지 실패)로 발견.

## 2026-07-21 — 슬랭 데이터 품질 보증 체계 설계 (진행 중)

**문제**: LLM이 생성한 슬랭 뜻/예문을 검증 없이 그대로 등록하면 부정확하거나 장난성 데이터가 섞일 위험이 큼.

**결정(안)**: 3단계 검증 절차 도입 예정 — ① 실제 관찰된 용례(source) 기록 필수, ② 최소 2개 독립 출처 간 의미 일치 확인, ③ verified 전환 전 블라인드 의미 테스트(제3자에게 예문의 슬랭을 가리고 뜻 추측시켜 일치율 확인). `schema.py`에 `verification_method` 필드를 추가하고, `review_status == "verified"`인데 이 필드가 비어 있으면 `loader.py`가 예외를 던지도록 강제할 예정.

**근거**: 이 프로젝트의 차별점은 "슬랭 데이터를 모았다"가 아니라 "노이즈 많은 사용자 생성 언어 데이터의 품질을 구조적으로 검증하는 파이프라인을 만들었다"에 있어야 함.

## 2026-07-21 — `poetry.lock` 제거, pip 기반 워크플로우로 통일

**문제**: `pyproject.toml`은 setuptools 기반 설치(`pip install -e .`)를 전제로 작성했는데, IDE가 자동 생성한 것으로 추정되는 `poetry.lock`이 커밋됨. `poetry.lock` 자체는 `.venv` 폴더와 달리 삭제 대상이 아니라 커밋해도 되는 파일(의존성 버전 고정 매니페스트)이지만, pip와 Poetry를 병행하면 설치 안내가 혼선을 줌.

**결정**: Poetry로 전환할 이유가 없다고 판단, `poetry.lock`을 `git rm`으로 완전히 제거하고 `.gitignore`에 추가해 재발 방지. pip 기반 워크플로우(`pip install -e .`, README의 설치 안내)로 통일.

**근거**: 포트폴리오 프로젝트 특성상 리뷰어가 Poetry를 별도로 설치하지 않고도 클론 후 바로 실행할 수 있어야 함. 재현성(버전 고정)이 꼭 필요해지면 그때 Poetry로 전환 검토.
